# script-author — 강사 대본 작성

대본의 **정본은 index.html 각 `<section>` 안의 `<aside class="notes">`** 다. 별도 문서로 대본을 만들면 화면과 어긋난다 — 문서가 필요하면 aside에서 추출해 생성한다. 발표 중에는 `S` 키(speaker view)로 본다.

## 형식 (설계안 §8 — 7항목)

```html
<aside class="notes">
  <p><b>역할</b> — 이 장표가 전체 강의에서 왜 필요한지 한 문장.</p>
  <p><b>대사</b> — 그대로 읽을 수 있는 완전한 문장들. 비개발자도 이해 가능하게.</p>
  <p><b>중요 문장</b> — 반드시 강조하고 잠시 멈출 문장. 화면의 핵심 문장(.slide__key)과 일치.</p>
  <p><b>오해</b> — 수강생이 하기 쉬운 잘못된 해석.</p>
  <p><b>교정</b> — 그 오해를 바로잡는 질문·예시.</p>
  <p><b>예상 질문</b> — Q: … / A: … (Java·엔진 내부 없이 답하는 기본 버전).</p>
  <p><b>시간</b> — 배정 N분. 부족할 때 생략 가능한 부분 / 절대 유지할 부분.</p>
</aside>
```

- 항목 출처: 역할·대사·시간은 `docs/course-plan.md`의 해당 장표(전달 메시지/대사, 배정 시간, 추가 이야기)에서 가져와 다듬는다. 오해·교정은 설계안의 "잘못된 해석" 단서 + DESIGN.md 오해 교정 원칙.
- fragment가 있는 장표는 대사 안에 **[다음]** 표시로 fragment 진행 시점을 적는다 — 말하는 순서와 화면 열리는 순서가 일치해야 한다(단계 노출 원칙).
- 예측 질문 패턴이면 대사에 "질문을 던지고 2~3명 답을 들은 뒤 [다음]"처럼 상호작용 지점을 명시한다.

## 톤·표현 규칙

- **비개발자 기준의 완전한 문장.** 코드·엔진 내부 구조 없이 설명한다. 개발 개념이 필요하면 비유를 먼저(예: "양식과 접수 건").
- 금지 표현과 배제 주제는 **COURSE.md §4**를 따른다("전문가"·"완전 정복"·"실행 엔진 숙달"·"Deep-Dive" 목표 용법, 근거 없는 단정, 범위 배제 주제).
- 용어는 **COURSE.md §7 용어 사전**의 표기를 따른다. 첫 등장에서 한 줄 풀이, 이후 같은 단어로. 사전에 없는 새 용어는 먼저 등재.
- 화면 텍스트를 그대로 낭독하는 대사를 쓰지 않는다 — 화면은 요지, 대사는 이야기. 둘이 같으면 대사를 다시 쓴다.
- 사실 주장은 `docs/verification-notes.md`와 일치해야 한다. 대사에서 새 사실이 필요해지면 먼저 검증한다.

## 절차

1. 대상 장표의 화면(HTML)이 먼저 완성되어 있어야 한다 — 대본은 화면 구조(fragment 순서·강조·핵심 문장)에 종속된다.
2. `docs/course-plan.md` 해당 항목과 화면 마크업을 나란히 놓고 7항목을 작성한다.
3. 시간 검산: 단원 내 장표들의 "배정 시간" 합이 교시(50분) 안에 드는지 확인하고, 초과분은 "시간" 항목의 생략 후보로 명시한다.
4. speaker view(`S`)로 열어 대사가 화면 진행과 맞는지 확인한다.

## 자가 점검

- 대사만 읽어도 강의가 진행되는가? (강사가 처음 보는 사람이어도)
- [다음] 위치가 fragment 수와 일치하는가?
- 중요 문장이 화면의 `.slide__key`와 같은 메시지인가?
- 금지 표현·미검증 사실이 없는가?
