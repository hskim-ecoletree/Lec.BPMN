# Workflow 모델링 관점의 BPMN·Camunda 7 교육 설계안

> 상태: PPT 제작 전 검토용 초안  
> 대상: OutSystems 교육을 거의 이수한 수강생. 개발 경력, Java, BPMN, 프로세스 엔진 경험은 전제하지 않는다.  
> 강사: Java 개발 10년, BPMN 기반 자동화·프로세스 자산화 애플리케이션 개발 3년  
> 과정: 09:00–18:00 전일 과정, 50분 수업 × 8교시 = 순수 교육 400분 + 점심 60분 + 휴식·전환·마무리 80분  
> 범위: BPMN과 Camunda 7을 이용해 Workflow를 구조적으로 생각하고 모델링하는 관점을 익힌다. OutSystems 제품 기능이나 Camunda 연동은 다루지 않는다.

---

## 1. 강의 목표 재정의

이 과정의 목표는 BPMN 전문가, BPM 컨설턴트, BPMS 운영자 또는 프로세스 자동화 전담자를 만드는 것이 아니다. 수강생이 교육 후 BPMN 관련 직종에 종사할 것이라는 전제도 두지 않는다.

이 과정은 BPMN을 **Workflow를 생각하고 설명하기 위한 여러 도구 중 하나**로 소개한다. 수강생이 업무 흐름을 볼 때 화면이나 기능 목록만 보는 데서 벗어나 다음을 질문할 수 있게 만드는 것이 목표다.

1. 이 Workflow는 무엇을 계기로 시작하고 어디에서 끝나는가?
2. 누가 어떤 일을 완료해야 하는가?
3. 어떤 조건 때문에 경로가 달라지는가?
4. 어디에서 사람·시간·메시지를 기다리는가?
5. 실패하거나 기한을 넘기면 어떤 경로로 복구하는가?
6. 이 흐름을 BPMN으로 표현하는 것이 실제로 도움이 되는가?

### 교육 후 기대 수준

수강생은 다음 정도를 수행할 수 있으면 충분하다.

- BPMN이 Flowchart, State Machine, Checklist, Case Management, 코드 기반 Orchestration과 어떻게 다른지 설명한다.
- 간단한 업무를 시작·역할·활동·분기·대기·종료·예외로 나누어 본다.
- BPMN의 핵심 요소를 보고 Workflow의 의미를 읽는다.
- XOR, Parallel, Inclusive, Event-based Gateway가 표현하는 업무 문장의 차이를 이해한다.
- Camunda 7의 토큰, Wait State, Job, Retry를 통해 모델이 실제로 어떻게 움직일지 상상한다.
- Camunda Modeler의 Properties를 전부 암기하지 않고, 실행 의미에 영향을 주는 값을 찾아 확인한다.
- BPMN을 써야 하는 경우와 쓰지 않는 편이 나은 경우를 구분한다.

### 명시적인 비목표

- BPMN 2.0 표준 전체 암기 또는 자격 취득
- BPM/BPMS 방법론 Deep-Dive
- Camunda 엔진 내부 코드, 데이터베이스 테이블, Java Delegate 구현 교육
- Camunda 7 운영 전문가 또는 마이그레이션 전문가 양성
- OutSystems와 Camunda의 연동·통합 설계
- 모든 Workflow를 실행 가능한 BPMN으로 자동화
- 기호를 많이 사용하거나 복잡한 모델을 만드는 능력

## 2. 강의의 핵심 메시지

강의 전체에서 다음 다섯 문장을 반복한다.

1. **BPMN은 목적이 아니라 Workflow를 명확하게 생각하기 위한 도구다.**
2. **좋은 모델은 도형이 많은 모델이 아니라 서로 다른 사람이 같은 흐름을 예상하게 하는 모델이다.**
3. **Gateway는 마름모가 아니라 ‘업무 경로를 선택하고 다시 모으는 규칙’이다.**
4. **Camunda는 이 과정의 최종 목적이 아니라 모델의 실행 의미를 관찰하는 확대경이다.**
5. **표현할 가치가 없는 세부사항과 BPMN에 맞지 않는 업무를 억지로 모델링하지 않는다.**

## 3. 스토리 기반 교육 설계 원칙

### 중심 사례: 신입사원 온보딩

BPMN 설명 파트는 고립된 도형 정의가 아니라 **한 사람의 입사 확정부터 첫 출근 준비 완료까지 이어지는 이야기**를 따라간다. 수강생은 먼저 업무 장면을 머릿속에 그리고, 그 장면을 더 정확하게 설명하기 위해 BPMN 요소를 하나씩 발견한다.

#### 등장인물과 시스템

- **신입사원 지민:** 입사 서류를 제출하고 안내를 기다린다.
- **HR 담당자 수진:** 온보딩 전체 흐름의 책임자다.
- **채용 매니저 민호:** 근무 형태, 장비, 접근 권한을 확인한다.
- **IT 담당자:** 계정, 장비, 보안 접근 권한을 준비한다.
- **시설 담당자:** 사무실 좌석과 출입카드를 준비한다.
- **외부 Identity Provider:** 계정 생성 API를 제공하며 일시적으로 실패할 수 있다.

#### 이야기의 전체 흐름

1. 지민이 입사 제안을 수락하면 `OfferAccepted` 메시지로 온보딩이 시작된다.
2. 수진은 입사 서류를 요청하고, 프로세스는 서류 도착 또는 제출 기한을 기다린다.
3. 서류가 도착하면 민호가 근무 형태와 필요한 장비·권한을 확인한다.
4. 원격/사무실 근무처럼 하나를 고르는 경로와, 역할에 따라 여러 접근 권한이 필요한 경로를 구분한다.
5. 계정 생성, 장비 준비, 교육 등록이 병렬로 진행된다.
6. 외부 계정 생성 API가 실패하면 Job Retry를 거쳐 필요 시 Incident가 발생한다.
7. 모든 필수 준비가 끝나고 입사일이 되면 첫 출근 준비 완료로 종료한다.
8. 서류 미제출, 채용 취소, 기술 실패는 서로 다른 경로와 복구 책임을 가진다.

#### 개념을 꺼내는 설명 순서

모든 BPMN 개념 장표는 가능한 한 다음 순서를 따른다.

1. **스토리 장면 제시:** 지금 지민·수진·민호에게 무슨 일이 일어났는지 말한다.
2. **수강생에게 예측 질문:** 다음에 누가 무엇을 해야 하며 무엇을 기다려야 하는지 묻는다.
3. **BPMN 표현 공개:** 수강생의 문장을 Event, Activity, Gateway, Flow로 바꾼다.
4. **Modeler 설정 확인:** 그 표현의 실행 의미를 바꾸는 Properties만 확인한다.
5. **토큰으로 재생:** 현재 위치와 개수, 대기 대상, 다음 Trigger를 따라간다.
6. **일반화:** 마지막에만 다른 Workflow에도 적용할 수 있는 개념 문장으로 정리한다.

#### 화면의 공통 시각 규칙

- BPMN 설명 장표 상단에는 항상 `현재 스토리 장면` 리본을 둔다.
- 등장인물의 색은 전 장표에서 유지한다: 지민=청록, 수진=파랑, 민호=주황, IT/시설=회색, 외부 시스템=빨강.
- 좌측 또는 중앙에는 이야기 속 Process 조각을, 우측에는 그 장면에서 발견한 개념과 Properties를 배치한다.
- 처음부터 완성 모델을 보여 주지 않고 이야기가 진행될 때마다 필요한 요소를 추가한다.
- 정의와 표는 사례를 해석한 뒤에 보여 준다. 정의를 먼저 읽고 예를 붙이는 순서를 피한다.
- 중심 사례로 설명하기 어색한 개념만 주문·환불·병원 접수 같은 짧은 보조 사례를 사용한다.

## 4. 기존 자료에서 바꿀 방향

- 완료 기준을 “토큰·분기·저장·복구를 전문적으로 설명한다”에서 “Workflow를 구조화하는 질문을 사용할 수 있다”로 바꾼다.
- BPMN의 요소와 Camunda Properties는 암기 대상이 아니라 모델의 의미를 확인하는 참고 정보로 다룬다.
- Gateway를 상세히 다루는 이유를 엔진 전문지식 습득이 아니라 **업무 문장의 모호함을 발견하는 훈련**으로 설명한다.
- BPMN 요소는 고립된 정의로 소개하지 않고 신입사원 온보딩 이야기에서 먼저 사건과 문제를 경험한 뒤 개념을 꺼낸다.
- Camunda 실행 설명은 엔진 구조 Deep-Dive가 아니라 “그림이 실제 상태와 대기로 바뀌면 어떤 일이 생기는가”를 이해하는 정도로 제한한다.
- 실습 평가 기준을 실행 가능한 완성 모델에서 문제 경계, 역할, 조건, 대기, 예외를 설명하는 능력으로 바꾼다.
- 최종 장표는 Camunda 용어가 아니라 실무에서 다시 사용할 Workflow 질문으로 마무리한다.

## 6. 장표별 구성안

모든 장표는 아래 항목을 동일하게 사용한다.

- **제목:** 화면에 표시할 직관적인 제목
- **핵심 내용:** 장표에 포함할 정보와 개념
- **스토리 장면:** 중심 사례에서 현재 일어나는 사건. 개념 설명보다 먼저 제시한다.
- **전달 메시지 / 대사:** 강사가 사실상 그대로 읽을 수 있는 핵심 문장
- **화면 구성 및 내용 배치:** PPT 구현 시 사용할 레이아웃과 시각 구조
- **연관 장표:** 앞뒤 흐름 또는 개념적으로 다시 참조할 장표 번호
- **중요도:** 5단계 우선순위 또는 실습 표시
- **배정 시간:** 설명·질문·짧은 활동을 포함한 대략적인 시간
- **추가 이야기:** 선택적으로 사용할 배경 지식, 유사 개념, 사담 또는 토론 거리

## 단원 1. 과정의 범위와 기대 수준 — 33분

BPMN 전문가 양성 과정이 아니라는 점과 수강생이 얻어갈 Workflow 관점을 명확히 한다.

### 01. Workflow를 구조적으로 생각하는 BPMN·Camunda 7 과정

- **제목:** Workflow를 구조적으로 생각하는 BPMN·Camunda 7 과정
- **핵심 내용:** BPMN은 전문 직종을 위한 종착점이 아니라 Workflow를 시작·역할·분기·대기·예외로 나누어 생각하게 해 주는 도구다.
- **스토리 장면:** 프롤로그. 아직 사례에 들어가지 않고 과정의 목적과 관점을 맞춘다.
- **전달 메시지 / 대사:** “오늘의 목표는 BPM 전문가 또는 BPMN 기호를 많이 아는 것이 아닙니다. BPMN이라는 도구의 존재와 BPMN 자체보다는 업무 흐름의 관점에서 업무를 어떻게 표현하고 업무 간의 흐름을 개괄할 수 있는지 알아보고, 필요할 때 BPMN을 사용할 수 있는 판단 기준을 얻는 것입니다.”
- **화면 구성 및 내용 배치:** 좌측 70%에 과정명과 한 문장 목표를 크게 배치한다. 우측에는 `8교시·교시당 50분`과 다섯 관점 `시작·역할·분기·대기·예외`만 세로로 둔다. 하단에는 대상과 비목표를 한 줄로 표시한다.
- **연관 장표:** 02, 04, 52
- **중요도:** ★★★★★ 최우선
- **배정 시간:** 3분

### 03. OutSystems와 BPMN은 Workflow를 어떻게 표현하는가

- **제목:** OutSystems와 BPMN은 Workflow를 어떻게 표현하는가
- **핵심 내용:** OutSystems와 BPMN/Camunda는 모두 흐름을 시각적으로 그리고 사람의 작업에 화면을 연결할 수 있다. OutSystems는 화면·데이터·로직과 Process를 애플리케이션 안에서 함께 구현하는 관점이 강하고, BPMN은 여러 역할과 시스템을 가로지르는 업무의 시작·작업·분기·대기·종료 의미를 표준 기호로 표현하는 관점이 강하다. `OutSystems Human Activity와 Screen` 및 `BPMN User Task와 Camunda Form`은 비슷한 업무 장면을 다루지만 같은 개념이나 1:1 대응 관계는 아니다.
- **스토리 장면:** 수진이 `입사 서류 검토` 업무를 수행하는 같은 장면을 좌측에는 OutSystems의 Process·Human Activity·Screen으로, 우측에는 BPMN User Task·Tasklist·Form으로 표현해 본다.
- **전달 메시지 / 대사:** “OutSystems를 배운 여러분에게 BPMN은 완전히 낯선 세계가 아닙니다. 둘 다 업무 흐름을 시각적으로 그리고, 사람이 수행할 단계에 화면을 연결할 수 있습니다. 다만 OutSystems가 화면·데이터·로직과 프로세스를 애플리케이션 안에서 구현하는 데 가깝다면, BPMN은 역할과 시스템을 가로지르는 업무의 시작, 대기, 분기, 종료 의미를 공통 언어로 설명하는 데 초점을 둡니다. 여기서 Screen이 곧 User Task라고 치환하면 안 됩니다. User Task는 사람이 완료해야 할 업무와 그 대기 상태를 뜻하고, Form은 그 업무를 수행하기 위한 사용자 접점 중 하나입니다. 두 도구의 연동을 배우는 것이 아니라, 같은 업무 장면을 서로 다른 관점으로 읽어 보는 것이 오늘의 연결 지점입니다.”
- **화면 구성 및 내용 배치:** 중앙에 같은 업무 장면 `입사 서류 검토`를 두고 2열 비교로 구성한다. 좌측에는 OutSystems 화면 예시 자리와 `Process → Human Activity → Screen`, 우측에는 Camunda Modeler 예시 자리와 `Process → User Task → Tasklist/Form`을 배치한다. 가운데 또는 하단의 비교표에는 `주된 관심: 앱 구현 / 업무 의미`, `사람의 작업: Human Activity / User Task`, `사용자 접점: Screen / Form`, `흐름 표현: OutSystems의 시각적 Flow / BPMN 표준 기호`를 넣는다. 최하단에는 `1:1 요소 대응 아님`, `OutSystems–Camunda 통합 수업 아님`, `BPMN User Task 자체가 화면은 아님` 세 문장을 주의 띠로 표시한다.
- **연관 장표:** 02, 04, 07, 14, 25, 40, 51
- **중요도:** ★★★★★ 최우선
- **배정 시간:** 10분
- **추가 이야기:** 강사가 OutSystems 기능을 자세히 설명하려고 하지 않는다. 먼저 수강생에게 자신들이 사용한 Workflow 편집 화면과 Human Activity에 연결한 Screen을 떠올리게 하고, 요소 이름이나 동작을 짧게 말해 달라고 요청한다. 그 답을 받아 BPMN의 Process·User Task·Form·Wait State 관점으로 번역한다. 잘못된 해석은 `OutSystems Process가 BPMN 구현이다`, `화면 하나가 User Task 하나와 같다`, `BPMN Form이 OutSystems UI 개발 도구를 대체한다`는 식의 1:1 치환이다.

## 단원 2. Workflow 도구 속 BPMN의 위치 — 47분

BPMN을 여러 Workflow 표현 도구 중 하나로 위치시키고, 시장의 부침과 AI 에이전트 시대의 가치를 과장 없이 연결한다.

### 06. Workflow란 무엇인가

- **제목:** Workflow란 무엇인가
- **핵심 내용:** Workflow는 특정 결과를 만들기 위해 일, 역할, 정보, 규칙, 시간이 연결되는 흐름이다.

- **전달 메시지 / 대사:** “Workflow는 자동화나 결재선만을 뜻하지 않습니다. 어떤 계기로 시작해 여러 사람과 시스템의 일을 거쳐 결과에 도달하는 전체 흐름입니다.”

- **연관 장표:** 05, 07, 08
- **중요도:** ★★★★☆ 핵심
- **배정 시간:** 7분
- **추가 이야기:** Recipe, 병원 접수, 배송 같은 비IT 예를 들어 Workflow가 소프트웨어보다 넓은 개념임을 보여 준다.

### 07. Workflow를 표현하는 도구들 사이에서 BPMN의 위치

- **제목:** Workflow를 표현하는 도구들 사이에서 BPMN의 위치

- **전달 메시지 / 대사:** “BPMN은 가장 고급인 도구가 아니라, 역할·메시지·대기·예외가 있는 반복 업무를 설명할 때 특히 유용한 선택지입니다.”

### 08. BPMN은 무엇인가

- **제목:** BPMN은 무엇인가
- **핵심 내용:** Business Process Model and Notation은 업무 흐름을 합의된 도형과 연결선으로 표현하는 표준 언어다.

### 09. BPMN과 BPMS가 한때 과도한 기대를 받은 이유

### 10. 부침 이후에도 BPMN이 남긴 실무 가치

- **제목:** 부침 이후에도 BPMN이 남긴 실무 가치
- **핵심 내용:** 여러 역할, 장기 대기, 반복되는 분기, 중요한 예외, 감사 필요성이 있는 안정된 Workflow에서 BPMN의 가치가 커진다.
- **스토리 장면:** 온보딩 사례가 여러 역할·장기 대기·반복 예외를 가지므로 BPMN에 비교적 잘 맞는지 평가한다.
- **전달 메시지 / 대사:** “BPMN을 모든 업무에 쓰지 않기 때문에 오히려 더 잘 쓸 수 있습니다. 책임과 대기가 중요한 반복 업무에 선택적으로 사용하면 됩니다.”
- **화면 구성 및 내용 배치:** 좌측 `잘 맞는 문제`, 우측 `다른 도구가 나은 문제`의 2열 비교로 구성한다. 하단에는 `적합성 우선`을 한 문장으로 배치한다.

- **배정 시간:** 7분

### 11. AI 에이전트 시대에 Workflow 모델이 주는 가치

- **제목:** AI 에이전트 시대에 Workflow 모델이 주는 가치
- **핵심 내용:** 에이전트가 Task 안의 판단을 유연하게 수행해도 사람의 검토, 권한, 시간, 감사, 실패 복구의 책임 경계는 남는다.

- **제목:** Intermediate Event로 대기와 통신을 표현하는 방법
- **핵심 내용:** Message Catch/Throw, Timer Catch, Signal은 현재 흐름이 무엇을 기다리거나 내보내는지 표현한다.

### 23. End Event와 Boundary Event로 결과와 예외를 표현하는 방법

- **핵심 내용:** End Event는 토큰의 결과를, Boundary Event는 Activity 수행 중 발생한 시간·오류 사건과 중단 여부를 표현한다.

- **배정 시간:** 5분
- **추가 이야기:** 실선·점선 Boundary의 차이는 도형 스타일이 아니라 원래 Activity 토큰을 유지하는지의 차이라고 강조한다.

### 24. Activity는 누군가 완료해야 하는 일을 표현한다

### 25. User Task의 책임과 완료 조건을 설정하는 방법

### 26. Service Task와 External Task의 실행 주체를 구분하는 방법

- **스토리 장면:** IT의 계정 생성이 외부 Identity Provider를 호출하며, 실행 주체와 실패 책임을 정해야 한다.

### 27. Subprocess와 Call Activity로 복잡도를 나누는 방법

### 28. Sequence Flow와 조건을 설정하는 방법

- **제목:** Sequence Flow와 조건을 설정하는 방법
- **핵심 내용:** Sequence Flow는 한 Process의 순서를 만들며 XOR/Inclusive의 조건과 Default는 연결선에 설정한다.
- **스토리 장면:** 근무 형태와 보안 검토 결과에 따라 나가는 Sequence Flow에 조건과 Default를 설정한다.

### 29. Pool·Lane·Message Flow·Data 요소를 필요한 만큼 사용하는 방법

- **제목:** Pool·Lane·Message Flow·Data 요소를 필요한 만큼 사용하는 방법
- **핵심 내용:** Pool은 Participant, Lane은 책임, Message Flow는 참여자 간 통신, Data Object는 정보 참조를 표현한다.
- **스토리 장면:** 회사, 지민, 외부 Identity Provider를 Participant로 나누고 역할별 Lane과 메시지를 배치한다.
- **전달 메시지 / 대사:** “책임과 통신을 보여 주는 요소는 중요하지만, Lane이 권한을 만들거나 Data Object가 실행 순서를 만드는 것은 아닙니다.”
- **화면 구성 및 내용 배치:** 상단 15%에 `현재 스토리 장면` 리본을 고정한다. 좌측 70%에 2개 Pool과 Lane, 그 사이 Message Flow 예를 두고 우측 30%에 Data Object와 주요 설정을 정리한다. 개념 정의와 일반화 문장은 사례를 읽은 뒤 우측 또는 하단에 공개한다.

## 단원 5. 분기와 병렬을 Workflow 언어로 이해하기 — 50분

Gateway를 상세히 다루는 이유는 전문지식 습득이 아니라 업무 문장의 `하나·모두·하나 이상·먼저 온 것`을 구분하기 위해서다.

### 30. Gateway는 일을 하지 않고 경로와 토큰을 제어한다

- **제목:** Gateway는 일을 하지 않고 경로와 토큰을 제어한다
- **핵심 내용:** Split은 토큰을 내보내는 규칙이고 Join은 토큰을 기다리는 규칙이다.
- **스토리 장면:** 민호의 확인 이후 하나를 고르는 경로와 여러 준비 작업을 동시에 시작하는 경로를 구분한다.
- **전달 메시지 / 대사:** “Gateway에 업무를 시키지 않겠습니다. 들어온 토큰 수, 나간 토큰 수, 선택 원인, 합류 대기만 설명하면 됩니다.”
- **화면 구성 및 내용 배치:** 상단 15%에 `현재 스토리 장면` 리본을 고정한다. 화면을 좌측 Split, 우측 Join으로 나누고 각각 `입력·출력·선택·대기` 네 질문을 배치한다. 토큰은 색 점으로 표시한다. 개념 정의와 일반화 문장은 사례를 읽은 뒤 우측 또는 하단에 공개한다.
- **연관 장표:** 12, 28, 31–37, 38
- **중요도:** ★★★★★ 최우선
- **배정 시간:** 4분
- **추가 이야기:** 마름모 안에 `승인 여부 판단` 같은 Task 이름을 쓰는 흔한 오해를 예로 든다.

### 31. Exclusive Gateway 분기: 여러 선택지 중 하나

- **제목:** Exclusive Gateway 분기: 여러 선택지 중 하나
- **핵심 내용:** 현재 데이터 조건을 평가해 정확히 한 outgoing Flow를 선택하며 조건은 상호 배타적이어야 한다.
- **스토리 장면:** 근무 형태가 `원격` 또는 `사무실` 중 하나로 결정되고 정확히 한 준비 경로가 선택된다.
- **전달 메시지 / 대사:** “XOR은 `이것 또는 저것 중 하나`입니다. 토큰 하나가 들어오고 원격 근무나 사무실 근무 중 조건에 맞는 한 경로만 나갑니다.”
- **화면 구성 및 내용 배치:** 상단 15%에 `현재 스토리 장면` 리본을 고정한다. 좌측에 원격/사무실 XOR 모델, 우측에 조건표와 Default Flow를 배치한다. 각 단계의 토큰 수 `1→1`을 크게 표시한다. 개념 정의와 일반화 문장은 사례를 읽은 뒤 우측 또는 하단에 공개한다.
- **연관 장표:** 28, 30, 32, 37, 48
- **중요도:** ★★★★★ 최우선
- **배정 시간:** 7분
- **추가 이야기:** 여러 조건이 동시에 참일 때 모델을 읽는 사람의 기대와 실제 선택이 달라질 수 있음을 토론한다.

### 32. Exclusive Gateway 합류: 실제로 도착한 경로를 통과

- **제목:** Exclusive Gateway 합류: 실제로 도착한 경로를 통과
- **핵심 내용:** XOR Merge는 다른 배타 경로를 기다리지 않고 실제로 도착한 토큰을 통과시킨다.
- **스토리 장면:** 원격 장비 배송 또는 사무실 좌석 준비 중 실제 도착한 한 경로가 공통 오리엔테이션 일정으로 합쳐진다.
- **전달 메시지 / 대사:** “원격 또는 사무실 경로 중 실제로 만들어진 토큰은 하나입니다. XOR 합류는 오지 않을 반대편 토큰을 기다리지 않습니다.”
- **화면 구성 및 내용 배치:** 상단 15%에 `현재 스토리 장면` 리본을 고정한다. 위·아래 두 경로 중 하나만 색을 활성화하고 중앙 Merge를 거쳐 다음 Task로 가는 2단계 그림을 사용한다. 개념 정의와 일반화 문장은 사례를 읽은 뒤 우측 또는 하단에 공개한다.
- **연관 장표:** 30, 31, 37
- **중요도:** ★★★★★ 최우선
- **배정 시간:** 5분
- **추가 이야기:** 단순 병합은 Gateway 없이 두 Sequence Flow를 다음 Task에 연결할 수도 있음을 보충한다.

### 33. Parallel Gateway 분기: 조건 없이 모든 경로

- **제목:** Parallel Gateway 분기: 조건 없이 모든 경로
- **핵심 내용:** Parallel Split은 조건을 보지 않고 모든 outgoing 경로에 토큰을 하나씩 만든다.
- **스토리 장면:** 계정 생성, 장비 준비, 필수 교육 등록이 조건 없이 모두 시작되어 토큰이 세 개가 된다.
- **전달 메시지 / 대사:** “Parallel의 업무 문장은 `조건과 상관없이 모두 한다`입니다. 토큰 하나가 들어오면 경로 수만큼 토큰이 생깁니다.”
- **화면 구성 및 내용 배치:** 상단 15%에 `현재 스토리 장면` 리본을 고정한다. 입력 토큰 하나가 재고·신용·주소 세 경로로 복제되는 장면을 크게 보여 주고 `1→3` 숫자를 강조한다. 개념 정의와 일반화 문장은 사례를 읽은 뒤 우측 또는 하단에 공개한다.
- **연관 장표:** 30, 34, 35, 37, 48
- **중요도:** ★★★★★ 최우선
- **배정 시간:** 7분
- **추가 이야기:** 논리적 병렬과 스레드·Worker의 물리적 동시성은 별개라는 설명을 반드시 포함한다.

### 34. Parallel Gateway 합류: 필요한 모든 토큰을 기다림

- **제목:** Parallel Gateway 합류: 필요한 모든 토큰을 기다림
- **핵심 내용:** Parallel Join은 각 incoming 경로의 필요한 토큰이 모두 도착할 때까지 먼저 온 토큰을 보관한다.
- **스토리 장면:** 세 준비 작업 중 먼저 끝난 것은 기다리고, 마지막 필수 작업까지 끝나야 `첫 출근 준비 완료`로 간다.
- **전달 메시지 / 대사:** “먼저 끝난 작업이 있어도 다음 단계로 가지 않습니다. 마지막 필요한 토큰이 도착해야 하나의 흐름으로 계속됩니다.”
- **화면 구성 및 내용 배치:** 상단 15%에 `현재 스토리 장면` 리본을 고정한다. 세 경로가 서로 다른 시각에 도착하는 3프레임 타임라인을 사용한다. 마지막 프레임에서만 다음 Task를 활성화한다. 개념 정의와 일반화 문장은 사례를 읽은 뒤 우측 또는 하단에 공개한다.
- **연관 장표:** 30, 33, 35, 37
- **중요도:** ★★★★★ 최우선
- **배정 시간:** 7분
- **추가 이야기:** 빠른 경로가 느린 경로를 기다리는 것이 업무적으로 맞는지 질문하면 동기화의 비용을 이해할 수 있다.

### 35. XOR 분기와 Parallel 합류가 멈출 수 있는 이유

- **제목:** XOR 분기와 Parallel 합류가 멈출 수 있는 이유
- **핵심 내용:** XOR은 하나만 만들었는데 Parallel Join이 모든 incoming 토큰을 기다리면 생성되지 않은 토큰 때문에 교착될 수 있다.
- **스토리 장면:** 원격/사무실 XOR 분기 뒤에 배송과 좌석 두 토큰을 모두 기다리는 Parallel Join을 붙인 잘못된 모델이 멈춘다.
- **전달 메시지 / 대사:** “선이 두 개라고 토큰이 두 개 있는 것은 아닙니다. 분기에서 실제 만든 토큰을 세지 않으면 멈추는 모델을 만들 수 있습니다.”
- **화면 구성 및 내용 배치:** 상단 15%에 `현재 스토리 장면` 리본을 고정한다. 좌측에 잘못된 모델, 우측에 토큰 원장처럼 `생성됨/생성되지 않음/기다림`을 표로 표시한다. 존재하지 않는 토큰 자리는 점선으로 그린다. 개념 정의와 일반화 문장은 사례를 읽은 뒤 우측 또는 하단에 공개한다.
- **연관 장표:** 31–34, 37, 48
- **중요도:** ★★★★★ 최우선
- **배정 시간:** 8분
- **추가 이야기:** 화면상 오류가 없는데 인스턴스가 진행하지 않는 모델 교착 사례를 강사의 실무 경험과 연결할 수 있다.

### 36. Inclusive와 Event-based Gateway가 필요한 상황

- **제목:** Inclusive와 Event-based Gateway가 필요한 상황
- **핵심 내용:** Inclusive는 조건에 맞는 하나 이상의 경로, Event-based는 미래에 먼저 도착한 사건 하나를 선택한다.
- **스토리 장면:** 역할에 따라 VPN·개발도구·관리자 권한 중 여러 개를 선택하고, 서류 제출 메시지와 기한 Timer 중 먼저 온 사건을 기다린다.
- **전달 메시지 / 대사:** “`하나 이상`과 `먼저 온 하나`는 전혀 다른 업무 문장입니다. 현재 조건을 보는지 미래 사건을 기다리는지 구분해야 합니다.”
- **화면 구성 및 내용 배치:** 상단 15%에 `현재 스토리 장면` 리본을 고정한다. 좌측 Inclusive 알림 채널 선택, 우측 Message/Timeout 경쟁을 대칭 배치한다. 아래에 `현재 데이터`와 `미래 사건`을 대비한다. 개념 정의와 일반화 문장은 사례를 읽은 뒤 우측 또는 하단에 공개한다.
- **연관 장표:** 30, 37, 44
- **중요도:** ★★★★☆ 핵심
- **배정 시간:** 7분
- **추가 이야기:** Inclusive Join의 활성 경로 계산은 복잡한 중첩 모델에서 설명과 운영이 어려워질 수 있음을 경고한다.

### 37. Gateway를 선택하는 네 문장과 합류 점검법

- **제목:** Gateway를 선택하는 네 문장과 합류 점검법
- **핵심 내용:** 정확히 하나=XOR, 모두=Parallel, 조건에 맞는 하나 이상=Inclusive, 먼저 온 사건=Event-based이며 합류 규칙은 별도로 판단한다.
- **스토리 장면:** 온보딩의 네 업무 문장을 XOR·Parallel·Inclusive·Event-based 선택 기준으로 다시 말한다.
- **전달 메시지 / 대사:** “Gateway 이름을 외우지 말고 업무 문장을 고르십시오. 그리고 분기에서 만든 토큰과 합류에서 기다릴 토큰을 다시 세십시오.”
- **화면 구성 및 내용 배치:** 상단 15%에 `현재 스토리 장면` 리본을 고정한다. 상단에 4개 선택 카드를 놓고 하단에 `분기 토큰 수`와 `합류 대기 대상` 두 개의 공통 질문을 크게 배치한다. 개념 정의와 일반화 문장은 사례를 읽은 뒤 우측 또는 하단에 공개한다.
- **연관 장표:** 30–36, 48, 52
- **중요도:** ★★★★★ 최우선
- **배정 시간:** 5분
- **추가 이야기:** 수강생에게 네 개의 짧은 업무 문장을 주고 Gateway 이름보다 선택 근거를 먼저 말하게 한다.

### 40. Wait State는 서버가 붙잡혀 있는 곳이 아니다

- **제목:** Wait State는 서버가 붙잡혀 있는 곳이 아니다
- **핵심 내용:** User Task, Message, Timer에서 상태를 저장하고 요청을 끝낸 뒤 새로운 Trigger가 오면 실행을 재개한다.
- **스토리 장면:** 프로세스가 지민의 서류와 입사일을 기다리며 상태를 저장하고, 새 Trigger가 오면 다시 움직인다.
- **전달 메시지 / 대사:** “며칠을 기다리는 Process가 며칠 동안 서버 요청을 열어 두는 것은 아닙니다. 상태를 남기고 잠들었다가 새 사건이 오면 다시 움직입니다.”

- **배정 시간:** 8분
- **추가 이야기:** 웹 요청, 스레드, Queue 대기와 Process 대기의 차이를 비개발자 수준으로 비유한다.

### 41. 저장 지점은 실패했을 때 되돌아갈 범위를 만든다

- **배정 시간:** 7분

### 42. Async·Job·Retry·Incident를 복구 경계로 이해하기

- **제목:** Async·Job·Retry·Incident를 복구 경계로 이해하기
- **핵심 내용:** Async는 사용자 응답과 후속 작업을 나누고 Job·Retry·Incident를 통해 실패를 운영 가능한 상태로 남긴다.
- **스토리 장면:** 계정 생성 작업을 async Job으로 분리하고 Identity Provider 실패를 Retry하다 Incident로 남긴다.
- **전달 메시지 / 대사:** “Async를 성능 체크박스로 보지 않겠습니다. 사용자에게 완료를 확정할 지점과 후속 실패를 다시 시도할 지점을 나누는 선택입니다.”

- **연관 장표:** 11, 26, 38, 41, 43, 45, 49
- **중요도:** ★★★★★ 최우선
- **배정 시간:** 8분
- **추가 이야기:** Retry는 멱등성이 없으면 외부 부작용을 반복할 수 있다는 짧은 사례를 포함한다.

### 43. 업무상 다른 결과와 기술 실패를 구분하는 방법

- **제목:** 업무상 다른 결과와 기술 실패를 구분하는 방법
- **핵심 내용:** 보안 검토 부적합·채용 취소 같은 업무 결과는 모델링된 경로로, 일시 기술 실패는 Retry로, 지속 실패는 Incident와 운영 개입으로 다룬다.

## 7. 실습 평가 기준

실습 결과는 BPMN 기호 수나 Camunda 실행 성공 여부로만 평가하지 않는다.

| 평가 항목 | 확인 질문 |
|---|---|
| 범위 | 시작과 종료가 한 문장으로 설명되는가? |

| 복구 | 실패 후 어디에서 어떤 조건으로 다시 진행하는가? |
| 적정 수준 | 중요하지 않은 구현 세부사항을 Process 밖에 둘 수 있는가? |
| 적합성 | 왜 BPMN을 사용했거나 사용하지 않았는지 설명하는가? |

1. **이 장표의 역할:** 전체 강의에서 왜 이 장표가 필요한지 설명
2. **그대로 읽는 대사:** 비개발자도 이해할 수 있는 완전한 문장
3. **중요 문장:** 반드시 강조하고 잠시 멈출 부분
4. **잘못된 해석:** 수강생이 흔히 할 수 있는 오해
5. **교정 방법:** 오해를 어떤 질문과 예로 바로잡을지 설명
6. **예상 질문과 답변:** Java 또는 엔진 내부 구조 없이 답하는 기본 버전
7. **시간 조정:** 시간이 부족할 때 생략 가능한 내용과 절대 유지할 내용

대본에서도 `전문가`, `완전 정복`, `실행 엔진 숙달`, `Deep-Dive` 같은 목표 표현을 사용하지 않는다. Gateway와 Camunda 동작은 Workflow를 정확히 읽는 데 필요한 만큼만 설명한다.
