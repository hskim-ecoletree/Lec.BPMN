오리엔테이션 · 도구 연결

OutSystems와 BPMN, 같은 장면의 두 표현

현재 스토리 장면 수진이 입사 서류를 검토합니다 — 같은 장면을 두 도구로 그려 봅니다.
OutSystems 11 · BPT BPMN · Camunda 7
사람의 작업
Human Activity↔User Task
자동 처리
Automatic Activity↔Service Task
사용자 접점
Destination Screen↔Form

다만 1:1 치환은 아닙니다 — Screen은 화면이고, User Task는 일과 그 대기 상태입니다.

OutSystems를 배운 여러분에게 BPMN은 낯선 세계가 아닙니다.
출처: OutSystems 11 — Notation Reference for BPMN Practitioners (success.outsystems.com)

대사 — BPT(Business Process Technology)는 OutSystems 11에서 프로세스를 앱 안에 내장해 실행하는 기능입니다. Human Activity가 실행되면 사용자의 Taskbox에 뜨고, Destination 속성이 가리키는 Screen에서 완료합니다. BPMN의 User Task는 사람이 완료해야 할 일과 그 대기 상태를 뜻하고, Form은 그 접점입니다. [다음]×3으로 대응 관계를 하나씩 확인.

오해 — "화면 하나 = User Task 하나"라는 치환. 교정 — User Task에 대응하는 것은 Human Activity이고, Screen/Form은 접점일 뿐. ODC에는 BPT가 없고 Workflows로 대체됨.

BPMN이라는 언어

지민의 온보딩 전체 이야기

현재 스토리 장면 입사 제안 수락부터 첫 출근 준비 완료까지, 전체 흐름을 처음 펼쳐 봅니다.
신입사원 온보딩 · Camunda 7 실행 모델
지민 · 신입사원 수진 · HR 민호 · 매니저 IT · 시설 외부 Identity Provider
지금은 훑어보기만 — 장면마다 하나씩 발견합니다
BPMN의 가치는 예쁜 그림이 아니라, 같은 실행을 예상하게 하는 데 있습니다.

대사 — 일반 문장이나 순서도로는 누가 책임지는지, 무엇을 기다리는지가 잘 드러나지 않습니다. BPMN은 역할(Lane)·메시지·대기·예외를 합의된 기호로 드러냅니다. 전체를 외우려 하지 마세요. 오늘 하루 동안 이 지도의 장면들을 하나씩 걸어갑니다.

분기 — 토큰으로 세는 법

토큰 — 진행 위치를 세는 도구

현재 스토리 장면 지민의 업무 한 건을 따라가며 토큰의 위치와 개수를 셉니다.
토큰 = 지금 일이 어디까지 왔는가
1 서류를 기다리는 중 제출 또는 3일 기한, 먼저 오는 쪽
3 병렬 준비 계정 · 장비 · 교육 — 1이 3이 된다
1 입사일 대기 3이 다시 1로 합류
0 첫 출근 준비 완료 토큰이 끝에 도달해 소멸
토큰 수를 세면 분기와 합류를 느낌이 아니라 규칙으로 설명할 수 있습니다.

대사 — 토큰을 DB 객체로 외우지 마세요. 지금 일이 어디까지 왔고 몇 갈래가 살아 있는지 설명하기 위한 사고 도구입니다. [다음] 서류 대기 1개 → [다음] 병렬 준비 3개 → [다음] 합류 후 1개 → [다음] 완료로 소멸.

분기 — 토큰으로 세는 법

XOR 분기와 Parallel 합류가 멈출 수 있는 이유

현재 스토리 장면 하나만 고르는 XOR 뒤에, 둘 다 기다리는 Parallel 합류를 붙였습니다.
안티패턴 · XOR Split → Parallel Join
원격 배송 — 토큰 생성됨 사무실 좌석 — 생성 안 됨 합류는 영원히 대기, 엔진은 오류로 감지하지 않음
‘선이 두 개’와 ‘토큰 두 개를 기다림’은 같은 말이 아닙니다.
출처: OMG BPMN 2.0.2 · Camunda 7 docs — parallel gateway (docs.camunda.org/manual/7.x)

대사 — [다음] XOR이 원격을 골라 토큰 하나가 생깁니다. [다음] 사무실 경로의 토큰은 처음부터 존재하지 않습니다. [다음] Parallel 합류는 모든 incoming에 토큰이 와야 열립니다. 오지 않을 토큰을 기다리며 업무 한 건(Process Instance)이 멈춘 채 살아 있습니다 — 화면상 오류가 없어서 더 위험합니다.

교정 — 분기에서 실제로 만든 토큰 수를 세게 한다. 합류는 분기 모양과 짝을 맞춘다.

실행과 복구

Async·Job·Retry·Incident — 복구의 경계

현재 스토리 장면 계정 생성이 외부 Identity Provider를 호출합니다. 일시 장애가 나면?
1 응답 확정 User Task 완료 시점에 Commit — 여기까지는 저장됨
→
2 Job 생성 asyncBefore가 계정 생성을 별도 Job으로 분리
→
3 Retry External Task는 Worker가 횟수·간격을 정함
→
4 Incident retries = 0 — 운영자가 원인 수정 후 재시도

Retry는 멱등(여러 번 실행해도 결과가 같음)이 아니면 외부 부작용을 반복합니다.

Async는 성능 옵션이 아니라, 완료를 확정할 지점을 나누는 설계입니다.
출처: Camunda 7 docs — job executor / external tasks / transactions (docs.camunda.org/manual/7.x)

대사 — [다음] 사용자에겐 완료를 확정해 줍니다. [다음] 후속 작업은 Job으로 분리되어 Job Executor가 실행합니다. [다음] 실패하면 재시도 — 일반 async Job은 엔진 기본 3회, External Task는 Worker가 handleFailure로 횟수와 간격을 직접 정합니다(예: 3회·5분). [다음] 소진되면 Incident가 남아 운영자가 개입합니다.

오해 — Async를 "빠르게 하는 옵션"으로 이해. 교정 — 응답 확정 지점과 실패 재시도 지점을 나누는 설계 선택.
