신입사원 온보딩
업무 흐름을구조로 읽는 법.
ecoletree

역할 — 첫 화면. 과정이 무엇을 가르치는지 한 문장으로 건다.

대사 — 오늘 하루, 업무를 화면과 기능의 목록이 아니라 '흐름의 구조'로 읽는 법을 익힙니다. 무엇으로 시작해 누가 무슨 일을 하고, 어디서 갈라지고 기다리고, 실패하면 어떻게 돌아오고, 어디서 끝나는가 — 이 짜임새가 구조입니다. 구조가 뭔지는 바로 다음 장, 지민의 온보딩에서 눈으로 보여드리겠습니다.

업무를 흐름으로 보는 눈
Q1–Q5

할 일 목록을 흐름으로 펼치는 순간,
시작·분기·대기·결과가 나타납니다.

수진의 할 일 — 화면·기능 목록 서류 요청 화면 서류 검토 화면 계정 생성 기능 장비 준비 화면 교육 등록 기능
같은 일 — 흐름의 구조
시작 일 분기 대기 결과

다이어그램이라서 구조인 것이 아닙니다 — 시작·분기·대기·결과를 답할 수 있는 배열이라서 구조입니다.

같은 다섯 가지 일입니다 — 배열이 달라지면 보이는 것이 달라집니다.

역할 — 스토리 개막. 표지가 건 '구조로 읽는 법'의 '구조'를 눈으로 해소한다.

대사 — 수진에게 지민의 입사 확정 메일이 도착합니다. 머릿속에 떠오르는 건 위와 같은 할 일 목록입니다 — 화면과 기능, 우리가 일해 온 방식이죠. 그런데 이 목록은 답하지 못합니다. 무엇을 계기로 시작하는지, 무엇을 기다려야 하는지. 아래는 같은 다섯 가지 일을 흐름으로 펼친 것입니다. [다음] 시작 — 이 업무 전체는 '입사 제안 수락'이라는 사건으로 시작됩니다. 목록에는 없던 정보입니다. [다음] 일 — 목록의 다섯 항목이 그대로 흐름 안에 있습니다. 화면이 아니라 '완료해야 할 일'로요. [다음] 분기 — 서류가 완전한지에 따라 길이 갈라져 요청으로 되돌아가기도 하고, 준비 셋은 동시에 갈라집니다. 목록은 되돌아가지 못하죠. [다음] 대기 — 서류가 올 때까지, 입사일이 될 때까지 흐름은 멈춰 기다립니다. [다음] 결과 — 끝은 항목을 다 지운 상태가 아니라 '첫 출근 준비 완료'라는 업무의 결과입니다. 이 짜임새 — 시작·일·분기·대기·결과 — 가 오늘 하루 계속 말할 '구조'입니다. [다음] 주의할 점 하나. 다이어그램으로 그렸다고 구조가 되는 것은 아닙니다. 네 가지를 답할 수 있는 배열이라야 구조입니다. 기호는 아직 외우지 않아도 됩니다.

오해 — "다이어그램으로 그리면 곧 구조다". 교정 — 그림이 아니라 시작·분기·대기·결과를 답할 수 있는 배열이 구조다.

오리엔테이션 — 도구 연결
Q6

OutSystems를 배운 여러분에게,
BPMN은 낯선 세계가 아닙니다.

OutSystems 11 · BPT BPMN · Camunda 7
사람의 작업
Human Activity↔User Task
자동 처리
Automatic Activity↔Service Task
사용자 접점
Destination Screen↔Form

다만 1:1 치환은 아닙니다 — Screen은 화면이고, User Task는 일과 그 대기 상태입니다.

BPT(Business Process Technology)는 OutSystems 11의 프로세스 기능입니다 — ODC에서는 Workflows로 대체됩니다.
OutSystems 11 — Notation Reference for BPMN Practitioners (success.outsystems.com)

대사 — 수진이 입사 서류를 검토하는 같은 장면을 두 도구로 그려 봅니다. [다음]×3 여러분이 쓰던 Human Activity가 BPMN의 User Task이고, Automatic Activity가 Service Task, Destination Screen이 Form의 자리입니다. [다음] 다만 치환은 아닙니다 — User Task는 화면이 아니라 "사람이 완료해야 할 일과 그 대기"입니다.

오해 — "화면 하나 = User Task 하나". 교정 — 대응되는 것은 Human Activity. Screen/Form은 접점일 뿐.

BPMN이라는 언어
Q1 · Q2

같은 그림을 본 사람들이, 같은 실행을 예상합니다.

지민 · 신입사원 수진 · HR 민호 · 매니저 IT · 시설 외부 Identity Provider
지금은 훑어보기만 — 장면마다 하나씩 발견합니다

대사 — 입사 제안 수락부터 첫 출근 준비 완료까지, 지민의 온보딩 전체 지도입니다. 일반 문장이나 순서도로는 누가 책임지는지, 무엇을 기다리는지가 드러나지 않습니다. BPMN은 역할과 대기와 예외를 합의된 기호로 드러냅니다. 외우지 마세요 — 오늘 하루 동안 이 지도를 장면 단위로 걸어갑니다.

분기 — 토큰으로 세는 법
Q3 · Q4

토큰을 세면, 실행이 보입니다.

1 서류를 기다리는 중 제출 또는 3일 기한, 먼저 오는 쪽
3 병렬 준비 계정 · 장비 · 교육 — 1이 3이 된다
1 입사일 대기 3이 다시 1로 합류
0 첫 출근 준비 완료 토큰이 끝에 도달해 소멸
토큰은 DB 객체가 아니라 사고 도구입니다 — 지금 일이 어디까지 왔고 몇 갈래가 살아 있는가.

대사 — 지민의 업무 한 건을 따라가며 토큰을 셉니다. [다음] 서류 대기 1개 → [다음] 병렬 준비 3개 → [다음] 합류 후 입사일 대기 1개 → [다음] 완료로 소멸. 위치와 개수만 세면 분기와 합류를 느낌이 아니라 규칙으로 설명할 수 있습니다.

분기 — 토큰으로 세는 법
Q3

‘선이 두 개’와 ‘토큰 두 개를 기다림’은
같은 말이 아닙니다.

안티패턴 · XOR Split → Parallel Join
원격 배송 — 토큰 생성됨 사무실 좌석 — 생성 안 됨 합류는 영원히 대기, 엔진은 오류로 감지하지 않음
XOR은 조건에 맞는 한 경로에만 토큰을 만듭니다. Parallel 합류는 모든 들어오는 선의 토큰을 기다립니다.
OMG BPMN 2.0.2 · Camunda 7 docs — parallel gateway (docs.camunda.org/manual/7.x)

대사 — 하나만 고르는 XOR 뒤에 둘 다 기다리는 Parallel 합류를 붙인 잘못된 모델입니다. [다음] XOR이 원격을 골라 토큰 하나가 생깁니다. [다음] 사무실 경로의 토큰은 처음부터 존재하지 않습니다. [다음] 합류는 오지 않을 토큰을 기다리며 업무 한 건(Process Instance)이 멈춘 채 살아 있습니다 — 화면상 오류가 없어서 더 위험합니다.

교정 — 분기에서 실제로 만든 토큰 수를 센다. 합류는 분기와 짝을 맞춘다.

실행과 복구
Q5

Async는 성능 옵션이 아니라,
완료를 확정할 지점을 나누는 설계입니다.

1 응답 확정 User Task 완료 시점에 Commit — 여기까지는 저장됨
→
2 Job 생성 asyncBefore가 계정 생성을 별도 Job으로 분리
→
3 Retry External Task는 Worker가 횟수·간격을 정함
→
4 Incident retries = 0 — 운영자가 원인 수정 후 재시도

Retry는 멱등(여러 번 실행해도 결과가 같음)이 아니면 외부 부작용을 반복합니다.

일반 async Job은 엔진 기본 3회 시도, External Task는 Worker가 handleFailure로 재시도를 정합니다.
Camunda 7 docs — job executor / external tasks / transactions (docs.camunda.org/manual/7.x)

대사 — 계정 생성이 외부 Identity Provider를 호출합니다. 일시 장애가 나면? [다음] 사용자에겐 완료를 확정해 줍니다. [다음] 후속 작업은 Job으로 분리되어 Job Executor가 실행합니다. [다음] 실패하면 재시도 — External Task는 Worker가 횟수와 간격을 직접 정합니다(예: 3회·5분). [다음] 소진되면 Incident가 남아 운영자가 개입합니다. [다음] 단, 멱등이 아니면 재시도가 부작용을 반복합니다.

오해 — Async를 "빠르게 하는 옵션"으로 이해. 교정 — 응답 확정 지점과 실패 재시도 지점을 나누는 설계 선택.
