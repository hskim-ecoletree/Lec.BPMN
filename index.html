오리엔테이션 · 도구 연결설계안 03

OutSystems와 BPMN은 Workflow를 어떻게 표현하는가

현재 스토리 장면 수진이 입사 서류 검토 업무를 수행하는 같은 장면을, 두 도구가 각자 어떻게 표현하는지 비교합니다.
OutSystems 11 · BPT
Process → Human Activity → Destination Screen

앱 안에 내장되어 바로 실행되는 프로세스. Human Activity는 Taskbox에 뜨고 Destination Screen에서 완료합니다.

BPMN · Camunda 7
Process → User Task → Tasklist / Form

역할·시스템을 가로지르는 업무 의미를 표준 기호로 표현. User Task는 사람의 일과 그 대기 상태를 뜻합니다.

관점OutSystems 11 (BPT)BPMN / Camunda 7
사람의 작업Human ActivityUser Task
자동 처리Automatic ActivityService / Script Task
사용자 접점Human Activity의 Destination ScreenUser Task의 Form
흐름 표기OutSystems 고유 flow 기호표준 BPMN 기호
요소가 1:1로 대응되지 않음 Screen ≠ User Task (대응은 Human Activity) ODC에서는 BPT 대신 Workflows
OutSystems를 배운 여러분에게 BPMN은 낯선 세계가 아닙니다. 같은 업무 장면을 서로 다른 관점으로 읽어 보는 것이 오늘의 연결 지점입니다.

BPMN 배경설계안 08

BPMN이란 무엇인가 — 지민의 온보딩 전체 이야기

현재 스토리 장면 입사 제안 수락부터 첫 출근 준비 완료까지, 온보딩 전체 흐름을 처음으로 펼쳐 봅니다.
신입사원 온보딩 · Camunda 7 실행 모델
지민 · 신입사원 수진 · HR 민호 · 매니저 IT · 시설 외부 IdP
같은 그림, 같은 예상

일반 문장이나 순서도로는 누가 책임지는지, 무엇을 기다리는지가 잘 드러나지 않습니다. BPMN은 역할(Lane)·메시지·대기·예외를 합의된 기호로 드러냅니다.

✎ 지금은 전체를 훑어보기만 합니다. 이후 장표에서 이야기가 진행될 때마다 필요한 요소를 하나씩 발견합니다.
BPMN의 핵심 가치는 예쁜 그림이 아니라, 같은 도형을 본 사람들이 같은 실행 흐름을 예상하게 만드는 데 있습니다.

토큰은 Workflow의 현재 진행 위치를 생각하는 도구다

현재 스토리 장면 지민의 업무 한 건이 흘러가는 동안, 토큰이 어디에 몇 개 있는지 장면마다 따라갑니다. (→ / Space 로 진행)
토큰 = 지금 일이 어디까지 왔는지 표시하는 사고 도구
지금 토큰은
  1. 1  서류 제출·기한을 기다리는 중
  2. 3  계정·장비·교육 병렬 준비 (1→3)
  3. 1  합류 후 입사일 대기 (3→1)
  4. ✓  첫 출근 준비 완료
› 토큰을 DB 객체로 외우지 마세요. 위치·개수·다음 Trigger를 설명하기 위한 개념 모델입니다.
토큰 수를 세면 분기와 합류를 ‘느낌’이 아니라 규칙으로 설명할 수 있습니다.
Gateway 집중설계안 35

XOR 분기와 Parallel 합류가 멈출 수 있는 이유

현재 스토리 장면 근무 형태를 하나만 고르는 XOR 뒤에, 배송·좌석 둘 다를 기다리는 Parallel 합류를 붙인 잘못된 모델입니다.
토큰 원장
경로토큰합류
원격 배송생성됨 ✓도착
사무실 좌석생성 안 됨 ✗영원히 대기
⛔ Parallel 합류는 모든 incoming에 토큰이 와야 진행합니다. XOR이 하나만 만들었으니 합류는 오지 않을 토큰을 계속 기다립니다.
✎ 엔진은 이를 오류로 감지하지 않습니다. 인스턴스는 멈춘 채로 계속 살아 있어, 화면상 오류 없이 진행만 안 됩니다.
안티패턴 · XOR Split → Parallel Join
‘선이 두 개’와 ‘토큰을 두 개 기다림’은 같은 말이 아닙니다. 분기에서 실제로 만든 토큰을 세지 않으면 멈추는 모델을 만들 수 있습니다.
