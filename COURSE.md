# COURSE.md — 과정 헌장

강의 **전체**의 목적·범위·준수/배제 사항의 단일 원천이다. 장표·대본·구성에 관한 모든 판단은 이 문서에 근거하며, 어긋나는 산출물은 리뷰에서 FAIL이다.

- `DESIGN.md`가 "어떻게 보이고 전달하나"(장표 수준)라면, 이 문서는 "무엇을 왜 가르치나"(과정 수준)다.
- 기존 초안(`docs/course-plan.md` 52장 설계안, 구 PPTX 58장)은 **정본이 아니라 참고 자료**다. 재구성(갈아엎기)은 이 헌장을 기준으로 한다.
- `[승인 필요]` 표시는 세션에서 아직 사용자가 확정하지 않은 가정이다. 확정되면 표시를 제거한다.

## 1. 목적 — 과정이 끝나면 수강생은

업무 흐름을 볼 때 화면·기능 목록이 아니라 다음 **여섯 질문**으로 구조를 읽을 수 있다.

| # | 질문 | 관점 |
|---|---|---|
| Q1 | 무엇을 계기로 시작하고 어디에서 끝나는가? | 시작·종료 |
| Q2 | 누가 어떤 일을 완료해야 하는가? | 역할 |
| Q3 | 어떤 조건 때문에 경로가 달라지는가? | 분기 |
| Q4 | 어디에서 사람·시간·메시지를 기다리는가? | 대기 |
| Q5 | 실패하거나 기한을 넘기면 어떤 경로로 복구하는가? | 예외·복구 |
| Q6 | 이 흐름을 BPMN으로 표현하는 것이 실제로 도움이 되는가? | 도구 적합성 |

**한 문장 요약**: BPMN은 목적이 아니라 Workflow를 명확하게 생각하기 위한 도구다.

모든 장표는 Q1~Q6 중 **최소 하나에 복무**해야 하며, 복무 대상은 `docs/inventory.md`에 기록한다. 어느 질문에도 복무하지 않는 장표는 만들지 않는다.

## 2. 대상과 전제

- **수강생**: OutSystems 교육을 거의 이수한 수강생. 개발 경력·Java·BPMN·프로세스 엔진 경험을 전제하지 않는다.
- **강사**: Java 10년, BPMN 기반 프로세스 애플리케이션 3년. BPMN 표준·OutSystems의 깊은 전문가가 아니다 — 이것이 "검증 노트에 없는 사실 주장 금지" 규칙의 이유다.
- **시간**: 09:00–18:00 전일, 50분 × 8교시 = 400분. 매 교시 마지막 2–3분은 핵심 회수. (교시 구조 확정)

## 3. 성공 기준 (기대 수준)

수강생이 다음을 할 수 있으면 성공이다. 실행 가능한 완성 모델을 만드는 능력이 아니다.

1. 간단한 업무를 시작·역할·활동·분기·대기·종료·예외로 나누어 말한다.
2. BPMN 핵심 요소를 보고 Workflow의 의미를 읽는다.
3. XOR·Parallel·Inclusive·Event-based가 표현하는 업무 문장의 차이를 구분한다.
4. 토큰·Wait State·Retry로 모델이 실제로 어떻게 움직일지 상상한다.
5. BPMN을 쓸 경우와 쓰지 않는 편이 나은 경우를 구분한다.

## 4. 비목표·배제 사항

### 과정 범위에서 배제 (다루지 않는다)
- BPMN 2.0 표준 전체 암기, 자격 취득
- BPM/BPMS 방법론 심화
- Camunda 엔진 내부 코드·DB 테이블·Java Delegate 구현, XML 구조
- Camunda 7 운영·마이그레이션 전문가 양성
- OutSystems 제품 기능 교육, OutSystems–Camunda 연동·통합
- 특정 벤더의 흥망사, 제품 비교
- 모든 업무의 BPMN 자동화 (도구 적합성 판단이 결론)

### 표현에서 배제 (화면·대본 공통)
- 이모지·픽토그램 글리프 (⛔ ✎ ✓ ✗ ↻ ① 등)
- 금지 표현: "전문가", "완전 정복", "실행 엔진 숙달", "Deep-Dive" — 목표·성취로 내세우는 용법 금지 (부정 문맥 "전문가가 될 필요는 없다"는 허용)
- 근거 없는 단정("항상", "무조건"), 검증 노트(`docs/verification-notes.md`)에 없는 사실 주장
- 번역체 문장, 설명 없는 축약어
- **강사·운영 컨텍스트의 화면 노출**: 교시 수·시간표·배정 시간·설계안 번호·수강 대상·전제 조건 등 과정 운영·제작 정보는 화면에 표시하지 않는다 — 수강생은 이미 자신이 누구인지 안다. 이런 정보의 자리: 제작 추적은 `data-plan` 속성(비가시), 시간·진행·대상 안내는 `aside.notes`(강사 전용), 대상·전제는 이 헌장 §2.

### 준수 사항 (사용자 제시 조건 — 집행 규칙은 DESIGN.md)

## 5. 핵심 메시지 — 전 과정에서 반복

1. BPMN은 목적이 아니라 Workflow를 명확하게 생각하기 위한 도구다.
2. 좋은 모델은 요소가 많은 모델이 아니라 서로 다른 사람이 같은 흐름을 예상하게 하는 모델이다.
3. Gateway는 마름모가 아니라 '업무 경로를 선택하고 다시 모으는 규칙'이다.
4. Camunda는 최종 목적이 아니라 모델의 실행 의미를 관찰하는 확대경이다.
5. 표현할 가치가 없는 세부사항과 BPMN에 맞지 않는 업무를 억지로 모델링하지 않는다.

각 교시 마무리 회수는 이 다섯 문장과 §1 여섯 질문 중에서 고른다.

## 7. 용어 사전 — 일관성 기준

같은 개념은 전 장표·대본에서 같은 표기로 쓴다. 첫 등장에서만 한 줄 풀이를 붙인다.

| 표기 | 첫 등장 풀이 | 혼용 금지 표기 |
|---|---|---|
| Workflow | 계기부터 결과까지 사람·시스템의 일이 이어지는 흐름 | 워크플로우, 업무 플로우 |
| 토큰 | 지금 일이 어디까지 왔고 몇 갈래인지 표시하는 사고 도구 | 실행 포인터, 실행 흐름(토큰 의미로) |
| User Task | 사람이 완료해야 할 일과 그 대기 상태 | 사용자 태스크, 휴먼 태스크 |
| Wait State | 상태를 저장하고 요청을 끝낸 뒤 트리거를 기다리는 지점 | 대기상태(붙여쓰기), 웨이트 |
| Process Instance | 실제 업무 한 건 | 인스턴스(첫 등장 단독), 프로세스 건 |
| Process Definition | 배포된 설계(버전 있음) | 프로세스 정의서 |
| Gateway | 경로를 선택하고 다시 모으는 규칙 | 게이트웨이(한글 표기) |
| Event | 흐름에서 무언가 일어나는 지점(계기·대기·결말) | 이벤트(한글 단독 표기) |
| Activity | 사람·시스템이 완료하는 일의 총칭(Task 포함) | 액티비티(한글 단독 표기) |
| Sequence Flow | 요소를 잇는 순서 연결(화살표) | 시퀀스 플로우, 순서 흐름(혼용) |
| Incident | 재시도 소진 후 운영 개입이 필요한 상태 | 인시던트, 장애(Incident 의미로) |
| Job | 엔진이 나중에 실행하려고 저장해 둔 비동기 작업 단위 | 잡, 배치 작업(Job 의미로) |
| External Task | 엔진이 실행하지 않고 외부 Worker가 가져가 처리하는 작업 방식 | 외부 태스크 |
| Worker | External Task를 가져가 실행하고 결과·실패를 보고하는 외부 프로그램 | 워커 |
| Business Key | 업무 인스턴스를 찾는 사용자 지정 식별자 | 비즈니스 키(첫 등장 이후 혼용 가능) |
| Service Task | 엔진이 자동으로 실행하는 작업 요소 | 서비스 태스크 |
| Automatic Activity | OutSystems 11 BPT의 자동 처리 요소 | 오토매틱 액티비티 |
| Destination Screen | OutSystems Human Activity가 가리키는 작업 화면 속성 | 목적지 화면 |
| Form | BPMN User Task의 사용자 입력 접점 | 폼(단독 첫 등장) |
| Human Activity | OutSystems 11 BPT의 사람 작업 요소 | Screen과 치환 금지 |
| BPT | OutSystems 11의 Business Process Technology | ODC 문맥에서 사용 금지(ODC는 Workflows) |
| Checklist | 할 일을 나열하고 완료를 표시하는 목록 — 대사에서는 '체크리스트'로 읽음 | 점검표 |
| Flowchart | 순서와 분기를 그리는 일반 순서도 — 대사에서는 '순서도'로 읽음 | 플로차트 |
| State Machine | 상태와 전이로 흐름을 적는 표기 — 대사에서는 '상태 기계'로 읽음 | 상태머신, 스테이트 머신 |
| XOR | Exclusive Gateway의 축약 표기 — 하나만 고르는 분기·합류 | 배타 게이트웨이 |
| Parallel | Parallel Gateway의 축약 표기 — 모두 실행하는 분기·합류 | 병렬 게이트웨이 |
| Async | 완료를 확정할 지점을 나누는 비동기 실행 설정(asyncBefore 등) | 비동기 처리(Async 의미로 단독 사용) |
| Retry | 실패한 Job을 다시 시도하는 것 | 리트라이, 재시도(Retry 표기 대신 단독 사용) |
| 멱등 | 여러 번 실행해도 결과가 같음 | idempotent 영문 단독 표기 |
| 안티패턴 | 문법상 가능하지만 의도와 다르게 움직이는 잘못된 모델 조합 | 앤티패턴 |
| AI 에이전트 | 목표를 주면 판단과 행동을 스스로 이어 가는 소프트웨어 | 에이전트(단독 첫 등장), AI봇 |

스토리 고유명(`OfferAccepted`, `DocumentsSubmitted`, `FirstDayReady`, 인물 이름)은 철자·표기를 바꾸지 않는다. 새 용어가 필요하면 **이 표에 먼저 등재**하고 쓴다.
